## Running the demo

All commands must be run from inside `subsuelo/` (the inner project root):

```bash
cd subsuelo
pip install -r requirements.txt
python build.py               # reproducible build of ALL regions + web assets (the canonical command)
python run_region.py spain    # one region only: peninsular Spain (IGME) ŌĆö raster + hotspots + Catastro drill
python run_region.py europe   # one region only: Variscan Europe (EGDI, 5 km) ŌĆö raster + hotspots + border-aware drill
python run_demo.py            # synthetic data
python run_live.py            # C├Īceres window: parcels-first + concessions friction
python run_ui.py --no-run     # serve the interactive Leaflet UI at http://127.0.0.1:8000
```

**Deploy** (the React app is a static site ŌĆö bundle + pre-built data, no server): `cd webapp && ./deploy/build-static.sh` ŌåÆ self-contained `webapp/dist/` (js/css/html + a real copy of `out/web/regions`, symlink dereferenced; vite `base:'./'` so it works at a domain root or subpath). Drop `dist/` on any static host (Netlify/Vercel/Cloudflare Pages/GitHub Pages/S3 ŌĆö see `webapp/DEPLOY.md`). Scheduled refresh: `DEPLOY_CMD="ŌĆ”" ./deploy/refresh.sh` runs `SUBSUELO_REFRESH=1 python build.py` ŌåÆ rebuild bundle ŌåÆ deploy.

## Reproducible builds / automatic updates

`build.py` is the single command that regenerates the whole page ŌĆö it fetches (or replays from cache) every data source, runs the deterministic transforms, and writes each region's web assets + a manifest, in one shot. Because the transforms carry no randomness, **a build is reproducible from its raw-download cache** (verified: an offline rebuild reproduces every artifact byte-for-byte).

- **`subsuelo/ingest/net.py`** ŌĆö every live fetch goes through `net.http_get(url, params, tag=ŌĆ”)`, a disk cache (`out/cache/`, keyed by url+sorted-params) that also appends to a `PROVENANCE` log (url, bytes, cache-hit, UTC). Atomic writes; a failed refresh falls back to the last cached copy (logged `stale`) so one flaky endpoint doesn't sink the build. Modes via env: `SUBSUELO_REFRESH=1` (refetch all + overwrite cache ŌĆö the **scheduled-update** command), `SUBSUELO_OFFLINE=1` (cache only, no network ŌĆö CI / source outage), `SUBSUELO_NO_CACHE=1`, `SUBSUELO_CACHE=<dir>`. (NE land/countries keep their own persistent geojson cache; idealista POST is unused.)
- **`build.py`** ŌĆö `python build.py [regionsŌĆ”]` (default all). Per region: resets provenance ŌåÆ `pipeline.run` (returns a source-count summary) ŌåÆ `export.export` ŌåÆ writes `regions/<key>/provenance.json` and stamps `built_at` + `sources` into that region's `meta.json`. Regions are isolated (one failing is recorded and skipped, leaving its old assets intact); writes `out/web/build.json` (UTC time, mode, per-region status + counts) and exits non-zero if any region failed. For a cron/CI auto-update: `SUBSUELO_REFRESH=1 python build.py`.
- Catastro ATOM feeds mix encodings (national office index is **latin-1**, per-province feeds are **utf-8**) ŌĆö `live._decode_atom` tries utf-8 then falls back to latin-1, so ├▒-provinces (Coru├▒a, Logro├▒o) resolve correctly.

## Modular region engine (add a new area in ~10 lines)

The scalable core is a **Region provider** pattern:
- `subsuelo/regions.py` ŌĆö the `Region` dataclass + `REGIONS` registry. Each area declares its grid (CRS, cell size, lon/lat window), its data-source callables (`granite`/`faults`/`occurrences`, each `bbox_4326 -> GeoDataFrame in region CRS`), WofE thresholds, and ŌĆö where an open cadastre exists ŌĆö `parcels_provider`/`concessions_provider`. Adding a country = one `Region(...)` entry.
- `subsuelo/pipeline.py` ŌĆö the region-agnostic engine `run(region)`: fetch evidence ŌåÆ rasterize ŌåÆ WofE (combined + per-metal) ŌåÆ hotspots (density-weighted; sharpened to real clusters) ŌåÆ optional parcel drill (top 8, on demand in the UI) ŌåÆ `./out` artifacts. `run_national.py`/`run_europe.py`/`run_region.py` are thin wrappers; `run_hotspots.py` is folded into the `spain` region's drill.
- `subsuelo/ingest/live.py` ŌĆö pluggable data-source functions (IGME, EGDI, Catastro, MITECO, IDECyL). `rasterize.py`/`score.parcels`/`live.py` read `config.*` dynamically so the engine can retarget the grid per region.

**`europe` now covers the whole continent** (widened from the Variscan-belt window `(-10,36,16,55)` to `(-10.5,35,30,68)` ŌĆö Iberia to the Baltics/Poland, Britain/Ireland to the Balkans, and the Fennoscandian Shield). This surfaced a real bug in `_egdi_wfs` (used by `egdi_granite`/`egdi_faults`/`egdi_felsic_volcanics`): **a single GetFeature request over a large bbox comes back geographically thinned server-side**, proportional to query area ŌĆö confirmed empirically by probing the exact same 10┬░├Ś10┬░ box (containing Cornwall's real, INSPIRE-mapped granite) at shrinking tile sizes: 10┬░ ŌåÆ 532 polys/0 in Cornwall, 5┬░ ŌåÆ 746/0, 3┬░ ŌåÆ 964/24, 2┬░ ŌåÆ 969/25 ŌĆö it plateaus at 3┬░. Pagination alone doesn't catch this (each page legitimately reports "no more results" well short of the true total ŌĆö confirmed the *pagination* itself is correct via a `resultType=hits`-adjacent exhaustion test), so `_egdi_wfs` now auto-tiles any bbox wider/taller than `_EGDI_TILE_DEG` (3.0┬░) into sub-tiles and concatenates (each tile independently cached by `net.http_get`, so a rebuild replays cheaply). This took the whole-continent granite fetch from 269 ŌåÆ 11,458 polygons and correctly recovered Cornwall's classic Sn-W granite. Hotspot tuning for the wider region: `target_pctl=94.0`/`top_k=22` empirically gives the best geographic spread (22/22 slots filled ŌĆö Iberia/Sweden/Czechia/France all distinctly represented; higher percentiles merge more into the #1 Iberia blob while dropping smaller real clusters below `min_cells` instead of shrinking them). `cell_size` bumped 5ŌåÆ7km (honest given 1:1M source + ~3x area). One real geological finding this surfaced: **Bergslagen (Sweden) and Cornwall initially showed identical rock-bottom posterior values** ŌĆö Bergslagen genuinely is skarn/VMS-hosted (not granite-related, correctly scored low), while Cornwall's absence was the tiling bug; after the fix Cornwall's real granite-hosted tin recovers and Sweden's Fennoscandian pegmatite belt (dominated by W, not Sn/Li) surfaces as hotspot #2. Each region's descriptive subtitle now comes from a single source of truth (`Region.label` ŌåÆ `commodities.json.region_description` ŌåÆ `meta.json` ŌåÆ `Sidebar.subtitleFor`) instead of hardcoded per-region-key strings in the frontend, which had gone stale twice already when a region's scope changed.

**Iberian Pyrite Belt (`iberia` region) ŌĆö a DIFFERENT mineral system (VMS base metals), same engine:** base metals (Cu/Zn/Pb/Ag) are dominantly volcanogenic massive sulfide, hosted by **felsic volcanics, not granite**, so they can't ride the granite model. The Region now carries its own commodity sets (`commodities`/`hotspot_commodities`/`proxy_commodities`, defaulting to the granite globals), and `iberia` overrides them with `config.VMS_COMMODITIES` = Cu/Zn/Pb/Ag + swaps the evidence: `granite=live.egdi_felsic_volcanics` (EGDI 1:1M lithology filtered to finegrainedigneous/pyroclastic/rhyolitoid ŌĆö the IPB Volcano-Sedimentary Complex), `occurrences=live.egdi_basemetals` (EGDI split into `_EGDI_GRANITE`/`_EGDI_VMS` so granite regions don't fetch base metals). Validated: WofE gives **positive weights for the volcanic host** (in-volcanics W+ 1.44, near-volcanics W+ 1.22) and ~neutral faults ŌĆö the VMS signature (stratabound, not vein) ŌĆö and the hotspots land squarely on the belt (Riotinto/Huelva, Aljustrel/Neves-Corvo, Tharsis). `pipeline.run`/`export` read `region.commodities` (not `config.COMMODITIES`) and glob `prospectivity_*.tif` per region so each region's assets carry only its own metals; the UI subtitle/legend/step-text derive their metal names from `commodity_models` (no hardcoded "Sn┬ĘW┬ĘLi"). `iberia` hotspots drill to Spanish parcels via `live.cadastre_spain` (Catastro INSPIRE CP WFS `ovc.catastro.meh.es/INSPIRE/wfsCP.aspx`, `cp:CadastralParcel`, urn:4326 lat,lon bbox ŌĆö GDAL flips; id `nationalCadastralReference`, area `areaValue` m┬▓; no cadastral value/municipio in INSPIRE CP) and to Portuguese parcels via `live.cadastre_portugal` (SNIC/DGT INSPIRE CP WFS `snicws.dgterritorio.gov.pt/geoserver/inspire/wfs`, `inspire:cadastralparcel`, id `nationalcadastralreference`, area `areavalue` m┬▓; CC BY 4.0) ŌĆö both wired into `_COUNTRY_CADASTRE`, so the Cu/Zn/Pb/Ag buy-list works over the whole IPB (Riotinto/Huelva in Spain, Aljustrel/Neves-Corvo in Portugal) and the europe-view Spanish/Portuguese hotspots drill too. **Portugal gotchas found by live-testing (undocumented anywhere):** `srsName=EPSG:4326` must be set explicitly on GetFeature or the geometry silently returns in an unlabelled Portuguese projected CRS with no error; the `bbox` param is lon,lat order with a plain `EPSG:4326` suffix (`minx,miny,maxx,maxy,EPSG:4326`) ŌĆö NOT lat,lon like Spain/Czechia's WFS 2.0 endpoints, despite superficially similar-looking services. Coverage is geographically lopsided ŌĆö dense in the south (Alentejo/IPB, exactly where `iberia`'s hotspots are: a nationwide `resultType=hits` query returned 1.79M parcels, and an IPB test box alone returned 400+) but sparse-to-empty in the north (the Sn-W-Li granite belt in `europe` ŌĆö Covas do Barroso lithium mine and central Panasqueira came back empty/patchy in testing), so it's only expected to help `iberia`, not `europe`'s Portugal hotspots. Both Catastro WFS's are slow (~1 min/box), so `iberia` uses a modest drill (`drill_radius_km=4`, `parcel_min_area_ha=2`); net.http_get caches the responses. **Remaining gap:** Spanish/Portuguese parcels have no real price (neither cadastre carries cadastral value, no DVF-equivalent), so the base-metal buy-list relies on the Eurostat/Destatis estimate like Czechia/Spain/Germany elsewhere.

**Spain** uses IGME 1:1M geology + national BDMIN; hotspots drill to Catastro parcels with legal-concession friction (IDECyL for Castilla y Le├│n via `config.CYL_PROVINCES`, else MITECO national). **Europe** uses EGDI (`geologicunitview` granite, `hike_all_faults_layer`; occurrences from the full INSPIRE inventory `egdi_mineraloccurrences_inspire` filtered server-side via an OGC `fes:Filter` to commodity Ōłł {Tin, Tungsten, Lithium, **Tantalum, Niobium**} ŌĆö the CRM subset layer had almost no tin, this gives ~1.3k Sn / 1.4k W / 120 Li / 80 Ta / 66 Nb). Europe hotspots **drill to per-country open cadastres** via a **border-aware** bbox provider (`live.parcels_for_bbox` ŌåÆ uses `live.natural_earth_countries()` to find every country the hotspot bbox overlaps and concatenates each one's cadastre: France IGN G├®oplateforme WFS `cadastre_france`, Czechia ─ī├ÜZK INSPIRE WFS `cadastre_czechia`, Germany (`cadastre_germany`, concatenating whichever of Saxony/NRW/Th├╝ringen return data ŌĆö see below), Spain `cadastre_spain`, Portugal `cadastre_portugal`; so a hotspot straddling the DE/CZ Erzgebirge pulls both. Portugal's own DGT/SNIC cadastre is wired in but its coverage is concentrated in the south, so Europe's northern Portugal hotspots ŌĆö the actual Sn-W-Li granite belt ŌĆö mostly stay raster-only regardless; Spain-in-EU-view drills via the same bbox `cadastre_spain` as `iberia`, distinct from the `spain` region's own municipio-ATOM drill). The `Region` carries either a `parcels_provider` (Spain, municipio-resolved) or a `parcels_bbox_provider` (Europe, a `drill_radius_km` box around each hotspot's representative point); `pipeline.drill_hotspot` branches on which is set. Hotspots correctly surface Iberia/Erzgebirge/Massif Central/Cornwall. **EGDI gotchas:** JSON output 500s (fetch GML); WFS 1.1.0 + EPSG:4326 returns lat,lon (`_egdi_wfs` flips to lon,lat; `egdi_occurrences` builds points from the `longitude`/`latitude` columns to sidestep it); `cql_filter` is ignored (use `fes:Filter`, no bbox param alongside it ŌĆö clip client-side). ## Web UI

- `subsuelo/web/export.py` ŌĆö reprojects `out/` artifacts to EPSG:4326 (Leaflet needs lat/lon), warps the raster to a colorized PNG overlay (alpha ramp: faint low ŌåÆ opaque high), writes `meta.json`, and emits `parcels_index.json` (the flat buy-list index for the budget search, with per-metal `prosp_<C>` sampled at each parcel centroid). **Masks sea cells to transparent** using Natural Earth 50m land (`live.natural_earth_land`, cached at `ingest/_ne_land_50m.geojson`) ŌĆö otherwise distance-to-granite/fault is nonzero over water near coastal features and the heatmap bleeds offshore.
- `subsuelo/web/index.html` ŌĆö self-contained Leaflet SPA: prospectivity heatmap + parcel-score choropleth + (live) BDMIN mineral-occurrence points colored by commodity, ranked-parcel sidebar, minerals panel, and a **target-commodity selector** (All/Sn/W/Li) that swaps the heatmap + re-ranks parcels per metal. Loads Leaflet + CARTO tiles from CDNs (needs internet).

**Per-commodity models:** the pipeline trains a separate WofE posterior for each of `config.COMMODITIES` = Sn/W/Li/**Ta**/**Nb**/**U**/**Mo**/**Be**/**Bi**/**Ba**/**As**/**Sb** ŌĆö all granite-related metals that share the same evidence (near-granite/near-fault/in-granite), so each extra one is near-free. U/Mo/Be/Bi are clean granite/greisen/pegmatite fits; **`config.PROXY_COMMODITIES` = {Ba, As, Sb}** are vein/fault hydrothermal ŌĆö the granite model is only a proxy, so their `commodity_models.reliable` is forced false (UI shows "indicative ŌĆö granite model is only a proxy", regardless of sample size). The base metals (Cu/Zn/Pb/Ag) are NOT in this set ŌĆö they're a different mineral system (VMS), handled by the separate **`iberia` region** (see below). The original set was Sn/W/Li/Ta/Nb (same evidence layers, deposits = showings whose `sustancia`/EGDI `commodity` mentions that metal), writing `prospectivity_<C>.tif` and per-parcel `score_<C>`/`prospectivity_<C>` columns. Ta/Nb are pegmatite by-products so they ride the same granite/pegmatite evidence "for free"; they're selectable per-metal layers but **`config.HOTSPOT_COMMODITIES` = Sn/W/Li** still drives hotspot detection (Ta/Nb shouldn't redraw where the clusters are). EGDI filters occurrences server-side by exact commodity (Europe: Ta 80 / Nb 66 reliable); IGME classifies from the Spanish `sustancia` string with a LiŌåÆWŌåÆSnŌåÆTaŌåÆNb priority (first match wins), but the per-metal training re-scans `sustancia` so Ta stays reliable in Spain (~42) even though most Sn-Ta showings take the Sn primary tag. `config`-level `MIN_TRAIN` (8) flags statistically thin models (Li has only 2 showings in the C├Īceres window ŌåÆ flagged "indicative"). Getting a reliable Li model needs a larger window (national scope) with more Li occurrences. Scoring precomputes parcelŌåÆcell membership once (`score.parcels.cell_membership`) so N rasters cost ~1├Ś the zonal loop.
- `run_ui.py` chains pipeline ŌåÆ export ŌåÆ static server; `--live` uses `run_live` instead of `run_demo`, `--no-run` reuses existing `out/`.

## Architecture

This is a Python geospatial pipeline for mineral prospectivity screening and land acquisition scoring in Extremadura, Spain (Sn-W-Li belt). The project lives in `subsuelo/subsuelo/` (inner package).

**Pipeline flow** (`run_demo.py` orchestrates):
1. **Ingest** (`ingest/`) ŌĆö generate or fetch evidence layers (geology contacts, faults, geochemistry, parcels, price listings)
2. **Model** (`model/wofe.py`) ŌĆö binarize evidence layers and run Weights of Evidence to produce a posterior probability raster
3. **Score** (`score/parcels.py`) ŌĆö compute per-parcel composite score via zonal stats

**Spatial context:** EPSG:25829 (ETRS89/UTM29N ŌĆö the correct zone for C├Īceres; was 25830/zone-30N, which put the window ~400 km east in the sea), 100├Ś100 km window around C├Īceres, 500 m cell resolution.

**Scoring formula** (from `config.py`):
```
score = 0.5 ┬Ę prospectivity_rank + 0.3 ┬Ę cheapness + 0.2 ┬Ę (1 ŌłÆ friction)
```

**Key files:**
- `subsuelo/config.py` ŌĆö central config: `BBOX`, `CRS`, `CELL_SIZE`, `SCORE_WEIGHTS`, `ENDPOINTS`
- `subsuelo/model/wofe.py` ŌĆö Agterberg/Bonham-Carter WofE with 0.5 continuity correction
- `subsuelo/ingest/synthetic.py` ŌĆö synthetic data generator (used by `run_demo.py`)
- `subsuelo/ingest/live.py` ŌĆö live ingestors, **validated 2026-07 against real services** (IGME ArcGIS REST + Catastro INSPIRE)
- `subsuelo/ingest/rasterize.py` ŌĆö turns live vector geology into the model's distance/mask grids (needs `scipy`)

## Status of live data ingestors

`ingest/live.py` is validated and driven by `run_live.py`:
- **IGME (ArcGIS REST, not WFS)** ŌĆö pinned services: `BasesDatos/IGME_BDMIN_Indicios` (mineral occurrences, layer 1), `Cartografia_Geologica/IGME_GeologicoCaceres_200` (layer 5 = contacts+faults via `TIPO`, layer 7 = lithology via `DLO`). Queried with `MapServer/<id>/query?f=geojson`, paged by `resultOffset` (maxRecordCount 1000). Occurrence commodity is classified from the Spanish `Sustancia` string (see `config.TARGET_COMMODITIES`, Li-first priority); granite host rock from `config.GRANITE_LITHOLOGIES`.
- **Catastro INSPIRE ATOM** ŌĆö per-municipio parcel GML zips at `.../CadastralParcels/{prov}/{code}-{NAME}/A.ES.SDGC.CP.{code}.zip`; the URL NAME must match the province ATOM feed exactly. `config.CATASTRO_MUNICIPIOS` defaults to four mineralized C├Īceres municipios. INSPIRE CP has no cadastral value, so `cadastral_eur_ha` is NaN. Requires Python `requests`+`certifi` (the Spanish gov TLS cert doesn't chain in curl's default bundle).
- **Catastro Minero legal concessions** (`live.mining_concessions`) ŌĆö the REAL legal mining-rights registry from the **MITECO national WFS** (`geoportal.minetur.gob.es/cgi-bin/mapservcm`, typename `derechosMineros`): concession POLYGONS with `nombre` + `leyenda` (type/status ŌåÆ granted `ŌĆ”O` / pending `ŌĆ”T` / lapsed `ŌĆ”CD`). GeoJSON output is broken server-side, so fetch **GML 3.1.1** and parse with GDAL; geometry is lon/lat despite the SRS label (force `set_crs(4326)`).
